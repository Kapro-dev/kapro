# v0.2.3 Scope Guardrails

`v0.2.3` stays in the pre-stable `0.x.x` release series. The milestone is for
delivery-promotion hardening, not broad cluster-management expansion.

## Ship List

| Area | Goal | Guardrail |
| --- | --- | --- |
| Two-phase staging | Make prepare, commit, and discard behavior observable and recoverable across spoke delivery. | Do not add a second delivery engine; harden the existing `internal/delivery` two-phase apply path. |
| CSR bootstrap | Improve short-lived bootstrap material, per-cluster identity, and renewal visibility. | Keep bootstrap scoped to Kapro delivery identity, not general cluster provisioning. |
| ClusterClassifier | Derive Kapro-owned labels that improve promotion safety and routing. | Do not copy broad cluster-inventory or policy engines. Classifiers must feed promotion decisions, bootstrap posture, drift, or adoption workflows. |

## ClusterClassifier Rule

`ClusterClassifier` is justified only when it makes delivery promotion safer or
easier to adopt. Valid uses include:

- deriving environment, region, substrate, or bootstrap labels used by Plans;
- flagging delivery readiness or drift posture for promotion gates;
- grouping existing GitOps clusters for observe-first adoption;
- explaining why a cluster is excluded from a promotion stage.

Invalid uses include:

- general Kubernetes inventory;
- generic policy distribution;
- whole-cluster lifecycle management;
- provider taxonomy that never affects a promotion, gate, rollout, rollback, or
  adoption path.

If a classifier field cannot name the promotion workflow it improves, it should
not ship in `v0.2.3`.

## Acceptance Bar

Every `v0.2.3` PR should include at least one of:

- focused unit tests for staging/bootstrap/classifier behavior;
- smoke or e2e coverage for a real adoption or spoke-bootstrap path;
- docs that convert an existing implemented path into a supported user workflow;
- metrics, events, or status conditions that make promotion recovery easier.

Do not open public CRD fields or SDK types for future aspirations. Preview
surfaces need runtime behavior, docs, tests, and an upgrade story in the same
milestone.
