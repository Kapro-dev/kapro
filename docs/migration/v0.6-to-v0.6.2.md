# v0.6 to v0.6.2 Manifest Migration

Kapro `v0.6.2` pulls the public-preview naming cleanup into the `v0.6.x`
line. This is a pre-stable breaking change so new users learn one clean YAML
shape before the public preview is announced.

Run the rewriter first, review the diff, then apply the migrated manifests:

```bash
# Preview one file on stdout.
kapro migrate v0.6 v0.6.2 path/to/manifest.yaml

# Rewrite one file in place.
kapro migrate v0.6 v0.6.2 --write path/to/manifest.yaml

# Rewrite a manifest directory recursively.
kapro migrate v0.6 v0.6.2 --write examples/my-fleet
```

For scripted pipelines, stdin and stdout are supported:

```bash
kapro migrate v0.6 v0.6.2 < old.yaml > new.yaml
```

## Rename Table

| v0.6 field | v0.6.2 field | Notes |
| --- | --- | --- |
| `Cluster.spec.substrate` | `Cluster.spec.delivery` | Delivery is the user intent; substrate is the selected implementation. |
| `Fleet.spec.substrate` | `Fleet.spec.delivery` | Same shape as `Cluster.spec.delivery`. |
| `spec.delivery.substrateRef` | `spec.delivery.ref` | The parent field already says delivery; the value is a bare local name. |
| `DeliveryUnit.spec.defaultFleetRef` | `DeliveryUnit.spec.defaultFleet` | Bare string refs do not carry the `Ref` suffix. |
| `DeliveryUnit.spec.defaultPlanRef` | `DeliveryUnit.spec.defaultPlan` | Same rule. |
| `Promotion.spec.deliveryUnitRef` | `Promotion.spec.unit` | Promotion action vocabulary uses short nouns. |
| `Promotion.spec.fleetRef` | `Promotion.spec.fleet` | Bare string local name. |
| `Promotion.spec.planRef` | `Promotion.spec.plan` | Bare string local name. |
| `PromotionRun.spec.deliveryUnitRef` | `PromotionRun.spec.unit` | Runtime API mirrors the action inputs for inspection. |
| `PromotionRun.spec.fleetRef` | `PromotionRun.spec.fleet` | Runtime object is controller-owned; do not commit it to Git. |
| `PromotionRun.spec.planRef` | `PromotionRun.spec.plan` | Runtime object is controller-owned. |
| `Trigger.spec.promotionTemplate.deliveryUnitRef` | `Trigger.spec.promotionTemplate.unit` | The trigger creates Promotion intent. |
| `Trigger.spec.promotionTemplate.fleetRef` | `Trigger.spec.promotionTemplate.fleet` | Bare string local name. |
| `Trigger.spec.promotionTemplate.planRef` | `Trigger.spec.promotionTemplate.plan` | Bare string local name. |
| `DeliveryUnit.spec.triggers[].fleetRef` | `DeliveryUnit.spec.triggers[].fleet` | Embedded trigger intent follows `Trigger`. |
| `DeliveryUnit.spec.triggers[].planRef` | `DeliveryUnit.spec.triggers[].plan` | Embedded trigger intent follows `Trigger`. |
| `Source.spec.substrateRef` | `Source.spec.substrate` | Bare string local name. |
| `SubstrateDiscoveryPolicy.spec.substrateRef` | `SubstrateDiscoveryPolicy.spec.substrate` | Bare string local name. |
| `*.spec.suspend` | `*.spec.suspended` | Kapro standardizes on the state adjective across preview APIs. |
| `Substrate.spec.discovery.enabled` | `Substrate.spec.discovery.suspended` | Boolean is inverted: `enabled: true` becomes `suspended: false`. |
| `Substrate.spec.substrate.kind` | `Substrate.spec.classRef.name` | Class selection now follows the Kubernetes class pattern. |
| `Substrate.spec.substrate.actuator` | removed | Use `classRef`, typed `configRef`, and controller-owned class status. |

Status-only fields such as `Target.spec.planRef` and
`Target.status.*.planRef` intentionally keep `planRef`. They identify a plan
node in a controller-owned execution graph, not a user-authored action operand.

## Before

```yaml
apiVersion: kapro.io/v1alpha1
kind: Fleet
metadata:
  name: checkout
spec:
  substrate:
    mode: pull
    substrateRef: flux
  clusters:
    - name: checkout-prod
      labels:
        kapro.io/stage: prod
---
apiVersion: kapro.io/v1alpha1
kind: Promotion
metadata:
  name: checkout-v1
spec:
  deliveryUnitRef: checkout
  fleetRef: checkout
  planRef: progressive
  version: v1.2.3
  suspend: true
```

## After

```yaml
apiVersion: kapro.io/v1alpha1
kind: Fleet
metadata:
  name: checkout
spec:
  delivery:
    mode: pull
    ref: flux
  clusters:
    - name: checkout-prod
      labels:
        kapro.io/stage: prod
---
apiVersion: kapro.io/v1alpha1
kind: Promotion
metadata:
  name: checkout-v1
spec:
  unit: checkout
  fleet: checkout
  plan: progressive
  version: v1.2.3
  suspended: true
```

## Validation Tightening

`Promotion.spec.version` and `Promotion.spec.versions` are now mutually
exclusive. Use `version` for one revision across the promoted unit set, or use
`versions` when each unit has its own revision:

```yaml
spec:
  unit: checkout
  fleet: checkout
  plan: progressive
  versions:
    api: v1.2.3
    worker: v4.5.6
```

The CRD rejects manifests that set both fields, and the controller refuses to
stamp a `PromotionRun` if an old manifest bypasses API-server validation.

## `suspended` Naming Decision

Kubernetes `CronJob` and Flux `Kustomization` use `spec.suspend`. Kapro keeps
`spec.suspended` in `v0.6.2` because the public-preview API already describes
object state across `Promotion`, `Fleet`, `DeliveryUnit`, `Trigger`, `Policy`,
`Cluster`, and discovery settings. The CLI still uses the familiar verbs:

```bash
kapro suspend checkout-v1
kapro resume checkout-v1
```

The dedicated field spelling is documented in
[API Naming](../concepts/api-naming.md).
