# v0.4/v0.5 Field Migration

Kapro v0.5.9 introduced backend-neutral delivery names while keeping the old
fields as compatibility aliases. Kapro v0.6 uses explicit substrate classes and
typed substrate config objects, so do not stop at the v0.5 intermediate shape
when preparing public-preview manifests.

## Backend

Old v0.4 shape:

```yaml
spec:
  driver: argo
  adapter: argo-cd
  runtime: Hub
```

v0.5 intermediate shape:

```yaml
spec:
  substrate:
    kind: argo
    actuator: argo
  execution:
    mode: hub-push
```

Current v0.6 shape:

```yaml
apiVersion: kapro.io/v1alpha1
kind: SubstrateClass
metadata:
  name: argo
spec:
  controllerName: kapro.io/argo
  executionModes:
    default: hub-push
---
apiVersion: argocd.substrate.kapro.io/v1alpha1
kind: ArgoCDSubstrateConfig
metadata:
  name: argo
spec:
  namespace: argocd
---
apiVersion: kapro.io/v1alpha1
kind: Substrate
metadata:
  name: argo
spec:
  classRef:
    name: argo
  configRef:
    apiVersion: argocd.substrate.kapro.io/v1alpha1
    kind: ArgoCDSubstrateConfig
    name: argo
  execution:
    mode: hub-push
```

Mapping:

| Old v0.4 | v0.5 intermediate | Current v0.6 |
| --- | --- | --- |
| `driver: argo` | `substrate.kind: argo` | `classRef.name: argo` |
| `driver: flux` | `substrate.kind: flux` | `classRef.name: flux` |
| `driver: oci` | `substrate.kind: oci` | `classRef.name: oci` |
| `adapter: argo-cd` | `substrate.actuator: argo` | `configRef.kind: ArgoCDSubstrateConfig` |
| `runtime: Hub` | `execution.mode: hub-push` | `execution.mode: hub-push` |
| `runtime: Spoke` | `execution.mode: spoke-pull` | `execution.mode: spoke-pull` |

Do not set multiple shapes on the same object. In v0.6 the v0.4 fields are
removed from the public CRD and the public examples use the current
`classRef`/`configRef` shape.

Run:

```bash
kapro doctor
```

For the 0.6 reset, follow [Pre-0.6 API Reset](pre-0.6-api-reset.md).

## Cluster Provider Kind

`Cluster.spec.provider.kind` now uses an open DNS-style string instead of a
closed enum. Existing values such as `outbound-agent`, `gcp-fleet`,
`gcp-connect-gateway`, `eks`, `aks-arc`, `rhacm`, `capi`, and `kubeconfig`
remain valid.
