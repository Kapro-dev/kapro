# kapro-cluster-controller Helm chart

This chart installs the spoke-side Kapro agent on a workload cluster. The agent
uses a bootstrap kubeconfig from the hub, registers or refreshes its `Cluster`
record, sends heartbeat leases, and reports delivery status for pull-mode
targets.

## Prerequisites

- A Kapro hub reachable from the spoke cluster.
- The hub operator installed with `cluster-bootstrap` in `controllers` and
  `hubAPIURL` set to the hub API server URL that spokes can reach.
- A bootstrap Secret generated on the hub with `kapro spoke bootstrap`.

Generate values and the Secret from the hub context:

```bash
kapro spoke bootstrap de-prod-01 \
  --hub-url https://hub.example.com:6443 \
  --secret-out /tmp/de-prod-01-bootstrap-secret.yaml \
  > /tmp/de-prod-01-values.yaml
```

Then switch to the spoke context and install:

```bash
kubectl create namespace kapro-system --dry-run=client -o yaml | kubectl apply -f -
kubectl apply -f /tmp/de-prod-01-bootstrap-secret.yaml

helm upgrade --install kapro-cluster-controller charts/kapro-cluster-controller \
  --namespace kapro-system \
  --create-namespace \
  -f /tmp/de-prod-01-values.yaml
```

For the full workflow and troubleshooting, see
[Registering a Cluster (Pull Mode)](https://github.com/Kapro-dev/kapro/blob/main/docs/cluster-bootstrap.md).

## Required Values

| Value | Meaning |
|---|---|
| `cluster.name` | Name of the hub-side `Cluster` object. |
| `hub.url` | Hub API server URL reachable from the spoke. |
| `bootstrap.kubeconfigSecret.name` | Secret containing the bootstrap kubeconfig generated by `kapro spoke bootstrap`. |

The CLI-generated values file fills these fields, plus the hub CA bundle when
needed.
